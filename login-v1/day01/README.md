# Day 01 — 「先跑起来再说」

## 场景

> 产品经理：「我们需要一个用户注册和登录功能，下周上线。」
> 你：「好的，一天搞定。」

## 你是怎么做的

用 Express + SQLite 搭了个简单的登录后端，功能能跑了：

- `POST /register` — 注册新用户
- `POST /login` — 用户登录
- `GET /users` — 查看所有用户（调试用，但忘删了）

你试了试，注册、登录都通，提交了 PR，感觉很棒。

---

## 今天的屎山预警

代码在当前看起来很清爽，但每一行都藏着后面要爆炸的种子：

### 🔴 密码明文存储

```javascript
// database.js
CREATE TABLE IF NOT EXISTS users (
  password TEXT      ← 密码直接明文存数据库！
)
```

这是**最有破坏力的安全隐患**。一旦数据库泄露，所有用户的密码直接暴露。

按理说应该 `bcrypt` 哈希加盐再存——但「先跑起来再说」的时候，谁在乎呢？

### 🔴 SQL 注入漏洞

```javascript
// main.js
db.prepare(`SELECT * FROM users WHERE username = '${username}'`)
```

字符串拼接 SQL，经典注入攻击面。

攻击者可以输入 `' OR 1=1 --` 绕过登录，或者 `'; DROP TABLE users --` 删库跑路。

应该用参数化查询 `WHERE username = ?`，但「先跑起来再说」嘛。

### 🔴 数据库路径硬编码

```javascript
const db = new Database('login.db');
```

文件路径写死在代码里。哪天换了环境或者用了不同的数据库来测试就会出问题。

### 🔴 没有输入校验

用户名和密码是**空的也能注册**。空字符串、超长字符串、特殊字符——全不拦。数据库层面可能会炸，但代码里没有处理。

### 🔴 没有错误处理

```javascript
db.prepare(`INSERT INTO users ...`).run();
```

没有 `try/catch`。数据库写入失败就直接进程崩溃。

### 🔴 HTTP 响应不合理

不管成功还是失败，状态码一律 `200`：

```json
{ "success": false, "message": "用户名或密码错误" }
```

真正的 API 应该返回 `400` / `401` / `409` 等语义化状态码。

### 🔴 调试接口未删除

```javascript
app.get('/users', (req, res) => { ... });
```

返回**所有用户的密码**。如果这个接口暴露到生产环境——灾难。

## 总结

> 功能能跑 ≠ 代码没问题。

今天的代码就是一个典型的**技术债起点**：
你选择了「最快路径」，代价是**安全、健壮性、可维护性**全部被牺牲掉了。

后面每一个新需求，都会在这个不牢靠的地基上继续堆叠——**屎山就是这样一天天长起来的。**
